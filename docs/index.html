    Grand Alliance, Light/Dark Theme & Mobile Image Positioning

    Three areas: faction selector supports selecting a Grand Alliance alone; light/dark theme toggle backed by CSS custom properties; and the fighter card image positioning workflow is redesigned for mobile — drag to pan, pinch to zoom, with a toggle overlay directly on the card.

    ✓ What Was Built

    • Grand Alliance-only selection: alliance headers in FactionSelect are now clickable buttons — clicking sets data.grandAlliance and clears data.faction/data.bladeborn. Card already renders only the GA runemark when faction is empty. Selected state highlighted in red; selectionLabel shows the alliance name alone.
    • Light/dark theme: data-theme attribute on <html> controls the active theme. Defaults to system prefers-color-scheme; manual toggle saves preference to localStorage and overrides the system setting permanently.
    • FOUC prevention: inline <script> in app.html reads localStorage (or system pref) and sets data-theme before the first paint.
    • CSS custom properties: :root defines dark-mode vars; [data-theme="light"] overrides them with warm parchment palette. Tailwind class overrides ([data-theme="light"] .bg-zinc-900 etc.) handle page-level layout colours without touching component class lists.
    • Light palette: shell #e4ddd3, inputs #ede8e2, text #2e2620 — warm parchment tones that complement the card aesthetic rather than generic grey-on-white.
    • Form components (FighterForm, AbilityForm, FactionSelect): all hardcoded hex values in <style> blocks replaced with var(--ui-*) CSS vars.
    • ThemeToggle: pill-shaped bordered button (sun/moon icon + "Light mode" / "Dark mode" label) placed inline in the landing page, below the card buttons. Not shown on editor pages.
    • Red button contrast fix: [data-theme="light"] .bg-red-900.text-white override keeps button text white on maroon backgrounds, counteracting the global text-white remap.
    • Contrast audit: --ui-text-subtle and [data-theme="light"] .text-zinc-500 darkened from #8a7e72 to #5a5048 (~5.3:1 on page bg, ~4.6:1 on alliance header bg — both pass WCAG AA).
    • Mobile image positioning — touch overlay: on the Preview tab, an "Adjust Image" toggle button appears when a model image is uploaded. When active, a transparent overlay covers the image area of the card; single-finger drag updates X/Y offset, two-finger pinch updates zoom. touch-action: none on the overlay suppresses browser scroll/zoom interference without needing preventDefault().
    • Positioning sliders hidden on mobile: Position X/Y/Zoom sliders in FighterForm are wrapped in hidden lg:block — desktop only. A hint line ("Adjust image position and zoom in the Preview tab.") takes their place on mobile, styled as plain text-sm text-zinc-300 with "Preview" bolded.
    • Below-card sliders removed: the intermediate slider strip below the card (added in an earlier iteration) is replaced entirely by the touch overlay. No sliders on the preview tab on mobile.
    • Range input thumb: custom CSS in app.css replaces the native thumb with a 20px circle (#991b1b) on both -webkit- and -moz-. Track uses var(--ui-surface-2) to adapt to both themes. Applies globally to all range inputs.
    • Hint label parity: "(left/right)" / "(down/up)" hints on slider labels use font-normal only (no hardcoded colour), inheriting var(--ui-field-label) from .sublabel — consistent with other form hints and theme-aware.
    • LAN testing: server: { host: true } added to vite.config.ts — dev server now listens on all network interfaces. Use the Network URL printed by Vite from any device on the same Wi-Fi. README updated with instructions.

    💡 Key Decisions

    Grand Alliance selection: new UI element or extend existing alliance headers?
    Extend existing headers — convert the sticky div.alliance-header to a button with onclick={() => select(alliance.label, '')}. Zero extra UI; the card already supports GA-only rendering when data.faction is empty.
    Theme implementation: Tailwind dark-mode class approach vs CSS custom properties?
    CSS vars + [data-theme] attribute overrides. All existing Tailwind classes are the dark values; switching to Tailwind's dark-mode system would require adding dark: prefixes everywhere. Instead, CSS overrides in app.css remap Tailwind's zinc/white classes under [data-theme="light"] — no component class changes needed. Form component <style> blocks use var(--ui-*) directly.
    Does system pref override manual toggle, or vice versa?
    Manual toggle wins and persists. On first visit (no localStorage) the system prefers-color-scheme is used. After a manual toggle the choice is saved to localStorage and sticks across visits. The page does not listen for live OS theme changes.
    Cookie/consent notice for localStorage?
    Not required. localStorage is purely local browser storage — it stores a single preference string, never leaves the device, is not sent with requests, and is not used for tracking. GDPR/ePrivacy cookie rules do not apply.
    Where should the theme toggle button live?
    Landing page only, inline in the content flow (below the card buttons). Setting a theme preference once on the home page and having it persist is sufficient — cluttering every editor page header is unnecessary.

    💡 Image Positioning Decisions

    Sliders in the preview tab vs touch-to-pan directly on the card?
    Touch overlay on the card (Option A). Sliders below the card are better than sliders hidden in the Edit tab, but still require the user to look away from the card while dragging. The overlay means the image and the interaction are the same object — direct manipulation. Sliders remain on desktop (in the form, side-by-side with the card).
    How to suppress scroll during touch interaction without preventDefault()?
    touch-action: none on the overlay div. Passive event listeners (Svelte's default) cannot call preventDefault(). touch-action: none achieves the same result — tells the browser to pass all touch events to JS and skip its own scroll/zoom handling — without needing non-passive listeners.
    Drag sensitivity: how many offset percentage points per pixel?
    50% per full image-area width/height in card space: sensX = 50/574, sensY = 50/503. Screen pixels are divided by cardScale first to get card-space pixels, then multiplied by sensitivity. Direction is inverted — drag right → image pans right → left side becomes visible → offset decreases.
    Range input thumb: custom CSS vs native?
    Custom CSS. Native thumbs on iOS/Android are ~20px but the hit target feels imprecise when fine-tuning. Explicit -webkit-appearance: none + 20px custom thumb with the project's red accent is consistent across platforms and matches the UI. Track colour uses var(--ui-surface-2) for theme compatibility.

    📱 Mobile Export Investigation

    PNG export on real iOS devices (iPhone) produced images with no background texture and intermittently no model image. A full switch to html2canvas was attempted but could not produce a correct result on either platform. All changes were reverted; desktop export uses dom-to-image-more unchanged.

    Root cause: iOS Safari + dom-to-image-more

    dom-to-image-more serialises the DOM into an SVG foreignObject, then draws it to canvas. iOS Safari refuses to render <img> elements and CSS background-image inside foreignObject. This is a fundamental browser security restriction, not a library bug. No workaround exists within dom-to-image-more.

    Attempted fix: html2canvas

    html2canvas uses Canvas 2D API directly, bypassing foreignObject. Both editor pages were switched to it, with additional canvas compositing to re-apply the SVG mask to the image area after capture.

    Problems encountered:

    • Model image overflow (desktop): .image-inner uses CSS mask-image for clipping with no overflow: hidden. Without the mask pass, transform: scale(zoom) on the model img caused it to overflow across the full card.
    • Background not rendering: background: url('/background.jpg') on .card never appeared in html2canvas output. Tried: CSS custom property as background-image, inline <img> at z-index:-1, programmatic inline style injection. None worked.
    • background-clip: text: .label-col uses this for the parchment-through-text effect on stat headers. Not supported by html2canvas.
    • Prerender 404: adding <img src="/background.jpg"> caused SvelteKit's static prerenderer to follow the src as a page route.

    Status: unresolved. All html2canvas code reverted. Mobile export replaced with an inline unavailable message on both editor pages and a note on the landing page. Known open issue.

    Multiple rounds produced no visual improvement. The background-image problem appears fundamental to how this project applies backgrounds (CSS background: on the card element), not a configuration issue. Further html2canvas attempts were deferred in favour of restoring a known-good working state.

    💡 Key Decisions

    Tab bar vs scroll vs drawer for mobile layout?
    Tabs. A scroll layout (form on top, card below) is awkward because the card is tall. A drawer adds complexity. Two equal-width tabs with a clear active state are the simplest and most native-feeling pattern.
    Why explicit-dimension wrapper instead of adjusting transform-origin?
    CSS transform doesn't affect layout — the element's layout box stays at 915px regardless of scale. Fighting this with transform-origin still leaves the card in an unpredictable position relative to flex siblings. Wrapping in a div with explicit computed dimensions (574×scale × 915×scale) is clean: flex and siblings see the correct visual size.
    Where should the export button live on mobile?
    Preview tab, above the card. On mobile the typical flow is: fill form → switch to preview → export. Keeping export in the Edit header would require switching back just to export. The preview panel is the natural place to export from.
    How to fix the dropdown on touch without a click-outside library?
    A $effect that watches showDropdown: when true, adds a document.addEventListener('click', close) via setTimeout(..., 0) to skip the opening click. Cleans up on effect teardown. No dependencies, no library.
    When should the side-by-side layout kick in — at 640px or 1024px?
    1024px (lg:). At 768px (iPad portrait) the form panel is 480px fixed, leaving only ~288px for the card preview — too cramped to be useful. Raising the breakpoint to 1024px means tablets in portrait get the full-screen tabbed layout, and the side-by-side layout starts at iPad landscape / desktop where there's enough room.

    Export fidelity — additional learnings

    border-top on child rows is not rendered by dom-to-image-more
    Per-row border-top: 1px solid black (inside a flex container) is silently dropped during dom-to-image-more serialization. Container-level borders work fine.     SVG icon width: auto is unreliable in export
    SVG elements with width: auto can render at an unexpected size inside dom-to-image-more's SVG foreignObject context — icons appeared smaller than in the browser preview. Fix: set explicit width: Npx; height: Npx on SVG elements (and resets on their wrapper spans).

    Setting base unconditionally would break the local dev server (links would point to /warcry-card-creator-2026/fighter which the dev server doesn't serve). Guarded by process.env.NODE_ENV === 'production' so dev works at localhost:5173 and the build works on GitHub Pages.

    User handles all git operations. Claude stages nothing and never runs git commit or git push.

    The editor needs ~1000px+ width to be usable (480px form panel + 574px card). Portrait tablets (e.g. iPad 768px wide) can't fit this. Landscape tablets (~1024px) can. The condition checks both width and orientation rather than a fixed breakpoint alone.

    Export fidelity — key learnings

    SVG mask-image URLs are not resolved by dom-to-image-more
    Relative url('/file.svg') in a CSS mask-image property is not fetched/inlined by dom-to-image-more during serialization — the masked element becomes invisible. Fix: import the SVG with Vite ?raw, build a data URL (data:image/svg+xml;charset=utf-8,${encodeURIComponent(raw)}), and apply it as an inline style attribute on the element. The file is kept in both static/ (URL serving) and src/lib/ (Vite import).
    .card * {} wildcard resets do NOT propagate through dom-to-image-more
    dom-to-image-more copies computed styles per-element as inline styles on the cloned DOM. It appears to skip "zero/default" values like border-width: 0 and background: transparent when they come from a wildcard ancestor selector. In the SVG foreignObject context the UA stylesheet then applies its own non-zero defaults (border, white background). Fix: add border: 0; outline: none; background: transparent directly to each element's own CSS rule — not via a wildcard. Elements with explicit non-transparent backgrounds (headers, badges) are unaffected.
    overflow: hidden on parent containers creates white-box artifacts in export
    Any element with overflow: hidden renders as an opaque white rectangle in dom-to-image-more's SVG foreignObject context, and its children inherit the white background. Fix: remove overflow: hidden from layout/table containers. For rounded-corner boxes that used it to clip inner backgrounds to the radius, apply border-radius directly to the inner header row (top corners) and last value row (bottom corners) instead. Keep overflow: hidden only on the root .card element.

    📐 Sizing & Spacing — The Long Road

    Most of this session was an iterative, sometimes messy process of extracting exact measurements from the v9 SVG reference and translating them into a faithful CSS layout. What looked simple turned into a long back-and-forth — with the user providing physical measurements in mm, then verifying by screenshot, then refining again.

    Physical dimensions and the 2× scale

    The card is 76×121mm physical. We render at 2× screen scale to get crisp PNG exports at double resolution. Conversion factor: 1mm = 7.559px (2 × 96 / 25.4). All measurements were converted once and kept as fixed pixel values — no CSS mm units, which behave unpredictably in browsers.

    Elementmmpx (2×)
    Card width76mm574px
    Card height121mm915px
    Image area (non-monster)52.6mm397px
    Image area (monster)28mm (reduced to fit)212px
    Table column width13.2mm100px
    Table row height (stats/weapons)8.5mm64px
    Damage bracket row height3mm23px
    Gap between tables / image3.9mm29px
    Parchment side/bottom padding~5mm / ~5mm38px

    Layout structure

    The card is a flex-direction: column container. The image section has a fixed height (flex: 0 0 397px); the parchment fills the rest (flex: 1). Inside the parchment, the fighter name uses flex: 1 to fill the space above the tables, which are anchored at the bottom with flex-shrink: 0.

    The tables are 5 columns × 100px = 500px wide (66mm), centred within the 574px card by the parchment padding.

    Monster layout

    When isMonster: true, the image section shrinks to 212px and a damage bracket table appends below the weapons box. The damage table is full-width (no border-radius, black borders, transparent background) and capped at 5 data rows to prevent overflow. The monster image height was reduced from the spec'd 37.9mm to ~28mm after overflow was observed in practice.

    Why it took so long

    Early attempts used percentage-based or approximate sizes — the card never matched the reference. Progress only clicked once every dimension was derived from the physical mm spec and consistently applied. Screenshots revealed mismatches multiple times before the spacing felt right. The table column and row sizes in particular required several passes: equal-width columns were not obvious from the SVG, and the gap between the image and first table was initially left unspecified then re-derived from the reference.
